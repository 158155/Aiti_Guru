# AITI Guru — Тестовое задание

Реализован REST API сервис для управления заказами покупателей.

## Спроектированная схема БД

Четыре связанные сущности:

- `categories` — дерево категорий с неограниченной вложенностью (самоссылка через `parent_id`)
- `products` — каталог товаров (наименование, остаток, цена, ссылка на категорию)
- `clients` — покупатели (наименование, адрес)
- `orders` + `order_items` — заказы покупателей с поддержкой разного набора товаров

Схема позволяет добавлять категории любого уровня вложенности без изменения структуры.

## SQL-запросы

Файл `queries.sql`:

- Сумма товаров по каждому клиенту
- Количество дочерних категорий первого уровня
- Выборка из view «Топ-5 самых покупаемых товаров за последний месяц»
- Анализ производительности и предложения по оптимизации (индексы, материализованное представление)

## Сервис (REST API)

Эндпоинт добавления товара в заказ:

```
POST /api/orders/{order_id}/items
```

Принимает ID заказа, ID товара и количество. Если товар уже есть в заказе — количество увеличивается. Если товара нет в наличии — возвращается ошибка 400.

Документация API доступна по адресу `/docs` (Swagger UI) или `/redoc` после запуска сервера.

## Конкурентный доступ

В решении реализована проверка остатка товара без блокировки строки (`SELECT ... FOR UPDATE`). При одновременных запросах одного товара из разных заказов возможна ситуация, когда суммарное количество заказанного превысит остаток на складе.

Для рельности это решается блокировкой строки при проверке:

```
product = db.query(Product).with_for_update().filter(Product.id == item_data.product_id).first()
if product.quantity < item_data.quantity:
    raise HTTPException(400, "Недостаточно товара")
product.quantity -= item_data.quantity
```

В рамках тестового задания требования к конкурентному доступу не были оговорены, поэтому реализована проверка без блокировки.

## Асинхронный подход

В решении задания использован синхронный SQLAlchemy ORM. При необходимости перехода на асинхронный стек:

- Добавить `asyncpg` как драйвер БД, заменить `create_engine` на `create_async_engine`
- `sessionmaker` заменить на `async_sessionmaker`, endpoint'ы переписать с `async def` и `await`
- В тестах использовать `pytest-asyncio` и `AsyncClient` вместо `TestClient`

Для текущего объёма задания синхронный подход не создаёт проблем с производительностью.

## Запуск

### Локальная разработка

1. Создать виртуальное окружение:

```
python -m venv venv
source venv/bin/activate
```

2. Установить зависимости:

```
pip install -r requirements.txt
```

3. Настроить подключение к базе данных:

```
cp .env.example .env
```

В файле `.env` указать строку подключения к PostgreSQL.

4. Создать схему базы данных:

```
psql -U postgres -d your_database -f schema.sql
```

Первичная миграция Alembic (`7611f7c997cc_инициализация_бд_из_schema_sql.py`) пустая — таблицы создаются в docker-compose через `schema.sql`. Вторая миграция (`706ebe472bc9_создание_представления_top5_products.py`) создаёт view `top5_products_last_month` с рекурсивным CTE для обхода иерархии категорий:

```
alembic upgrade head
```

5. Загрузить тестовые данные (категории, 100 товаров, 10 клиентов, 50 заказов):

```
python scripts/create_test_data.py
```

6. Запустить сервер:

```
uvicorn app.main:app --reload
```

Сервер будет доступен на `http://localhost:8000`.

### Docker

```
docker-compose up -d
```

Запустятся PostgreSQL и приложение. Документация API — `http://localhost:8000/docs`.

## Тесты

```
pytest tests/ -v
```

19 тестов, покрывающих API заказов, ограничения БД, представление top5 и интеграционные сценарии.
